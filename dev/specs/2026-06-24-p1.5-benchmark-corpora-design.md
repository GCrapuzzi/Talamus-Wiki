# P1.5 — Benchmark corpora & speed (Round 1) — Design

- **Date:** 2026-06-24
- **Status:** approved (brainstorm), pre-implementation
- **Phase:** P1.5 in [dev/ROADMAP.md](../ROADMAP.md)
- **Branch:** `feat/p1.5-benchmark-corpora`

## Decisions recorded (the 🧠 gate)

- **Third corpus = a third LOCAL enriched brain** (not a public judged set), because
  it uniquely validates the full extract→enrich→ontology workflow (today only the
  book does) and is €0 / offline / CI-safe.
- **Source = curated original factual articles (CC0)** authored by us — clean to
  commit, controllable, and multi-domain (anti-overfit). A public set
  (Wikipedia/Gutenberg) was rejected for round 1 (license/attribution/commit-bulk).
- **Scope = everything in one round:** build the third corpus, extend the existing
  corpora (negatives + judged queries), and audit/lock the FAST/HEAVY tiers.

## Goal & why

Make the ablations that gate retrieval/quality work (P2, P3, P5) trustworthy, while
keeping every local and CI run fast. Today there are two corpora (docs 120-case,
book 35-query); only the book is enriched, and the docs eval has only 10 negatives
(honest-refusal is a 1-question delta). The two-corpora law only protects quality if
the corpora carry signal across domains.

## Current state (verified)

- **Docs corpus** `examples/eval-cases-real.json`: 120 judged cases — direct 30,
  vague 30, cross-source 20, temporal 15, code 15, **negative 10**. Built
  deterministically (no LLM) by `corpus.build_docs_corpus` from frozen
  `tests/fixtures/docs-corpus/*.md`.
- **FAST tier already exists:** `tests/test_talamus_recall_floor.py` re-runs the
  120-case eval inside `python dev.py` and asserts recall@5 ≥ 0.45, MRR ≥ 0.40,
  hit ≥ 0.55.
- **HEAVY tier already exists:** `TALAMUS_BENCH_HEAVY` gates
  `test_benchmarks_{adaptive_floor,beir_datasets,mem0,vectordb}` (download/LLM/scale).
- **Book corpus** (35-query, local/copyright-safe) drives the shootout
  (`benchmarks/shootout`, `benchmarks/ask_eval`); the only enriched corpus.

So P1.5 round 1 is **extend + build the third corpus + lock**, not build-from-scratch.

## Component A — The "garden" corpus (third enriched brain)

**Source (committed, frozen fixture):** `tests/fixtures/garden-corpus/*.md` — ~18
short, original, factual articles (CC0), across **6 unrelated domains × 3**:
cooking, astronomy, law, history, biology, personal finance. Each file is one
self-contained mini-article (~150–350 words), plain Markdown with an H1 title and
1–3 `##` sections. Frozen like `docs-corpus` (never edited once judged).

**Deterministic build (no LLM):** add `build_garden_corpus(paths, repo_root)` to
`corpus.py`, mirroring `build_docs_corpus` (one note per `##` section + a doc-level
note; real provenance; part-of relations). This is the CI-safe corpus the FAST tier
uses.

**Judged eval:** `examples/eval-cases-garden.json` — judged queries across the
domains: direct, vague, **cross-domain negatives** (a cooking question must not pull
law/biology notes — strong, natural negatives), and a few cross-source. ~30–40
cases, with a healthy negative share (≥ 10).

**FAST tier test:** `tests/test_talamus_recall_floor_garden.py` — deterministic
recall floor on the garden corpus (anti-overfit for the retrieval algorithm; runs
every commit). Floors set from the first measured run minus a safety margin.

**HEAVY tier test:** `tests/test_benchmarks_garden_enrich.py`
(`@unittest.skipUnless(TALAMUS_BENCH_HEAVY)`) — builds the garden brain with the
**real** extract+enrich+ontology pipeline via a local LLM (gemma) and asserts:
enrich writes symptom vocabulary, ontology induces ≥1 candidate type, and vague-query
retrieval improves vs the un-enriched build. This validates enrich/ontology on a
diverse corpus, beyond the book.

## Component B — Extend the existing corpora

- **Docs negatives:** grow `examples/eval-cases-real.json` negatives from 10 → ~25
  (out-of-domain questions; safe — negatives carry an empty `relevant[]`, independent
  of the frozen doc titles). Makes negative-rejection statistically meaningful.
- **A few more judged queries** (vague / cross-source) referring only to existing
  frozen doc content.
- Keep the positive-side recall floors stable; negatives don't change recall@k of
  positives but make `negative_rejection` a real signal.

## Component C — Tiering audit & lock

- **Measure** the FAST tier runtime: the recall floors (docs + garden) inside
  `python dev.py` must stay in **seconds**. Audit `benchmarks/run.py` and the heavy
  tests; confirm nothing slow leaks into the gate.
- **Document** the FAST vs HEAVY contract in `benchmarks/README.md`: FAST = recall
  floors in the gate (seconds, no network/LLM); HEAVY = `TALAMUS_BENCH_HEAVY`
  (download / LLM / scale, on-demand).

## Testing

- New: `test_talamus_recall_floor_garden` (fast), `test_benchmarks_garden_enrich`
  (heavy/gemma).
- Updated: `eval-cases-real.json` (negatives); `test_talamus_recall_floor` only if
  the distribution shift requires it (it should not — positives unchanged).
- `python dev.py` ALL GREEN; FAST-tier addition measured and reported in seconds.

## Exit criteria (from roadmap P1.5)

The fast tier runs in seconds (measured) here and in CI; a third corpus exists;
ablations on the extended corpora show stable signal across runs; heavy runs are
gated and documented as on-demand; gate green.

## Out of scope (round 1)

- A frozen **post-enrich** snapshot of the garden corpus for the FAST tier (enrich
  validation stays heavy-tier for now; the fast tier validates retrieval on the
  deterministic build).
- A public judged retrieval set (Wikipedia/Gutenberg) — deferred.
- UI / shootout changes; the book corpus stays as-is.
